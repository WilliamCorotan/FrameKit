# Document queries and pagination

Framekit pushes supported document list operations into the database. Postgres applies filters, case-insensitive substring search across top-level field values, projection, sorting, offset, cursor boundaries, and the bounded limit before returning rows. Values are bound parameters; field names are validated against DocType metadata and are also passed as SQL parameters rather than interpolated into query text.

Supported filters are scalar equality, arrays (`in`), and the `eq`, `ne`, `in`, `contains`, `gt`, `gte`, `lt`, `lte`, and `isNull` operators. Numeric and currency comparisons use numeric ordering; other field types use deterministic text ordering. JSON fields can be filtered for equality/null but cannot be sorted. Unknown fields/operators, JSON sorts, malformed cursors, limits outside `1..1000`, and negative offsets fail with a `422` error instead of falling back to an in-process scan.

## Stable cursors

List responses remain JSON arrays. When another page exists, HTTP responses include `x-next-cursor`; the SDK `listPage()` method returns it as `nextCursor`. Pass that opaque value back as `cursor` while retaining the same sort.

Cursors are versioned base64url payloads generated by Framekit. They capture the sort field, direction, last sort value, and document ID. Ordering always uses document ID ascending as the final tie-breaker, including when the primary field is descending. The keyset boundary is stable when rows are inserted before a page while it is being traversed and does not depend on the cursor row continuing to exist. As with other keyset systems, changing a record's active sort value during traversal can move it across the boundary.

Do not parse, construct, or persist assumptions about cursor contents. A cursor is rejected if it is malformed or reused with a different sort contract.
