# Exact decimals, computed fields, and validators

Framekit has one public contract for `decimal` and `currency`: a base-10 JSON string with an optional leading minus, no leading plus, no exponent, no leading zeroes, and no more fractional digits than the field scale. Declare `precision` and `scale` on the field; defaults are `(18, 6)` for decimal and `(18, 2)` for currency. JavaScript numbers and bigints are rejected at every public write boundary because their representation or serialization cannot guarantee the same contract. Runtime responses, Postgres JSONB rows, outbox events, replayed responses, generated OpenAPI schemas, and generated SDKs all preserve strings.

Computed fields use a strict, bounded declarative contract: `sum`, `subtract`, and `multiply` operate on exact decimal/currency dependencies, while `concat` operates on textual dependencies. Metadata rejects irrelevant or unknown keys, unknown dependencies, invalid types, duplicate/contradictory validators, and cycles. Values are evaluated in dependency order, so multi-level computed DAGs are supported. Arithmetic rescales integer coefficients exactly; a non-zero remainder is rejected instead of rounded. Computed values are recomputed on every write and cannot be supplied by clients, even when the supplied value matches the result.

Validators are declarative and return stable violation codes in `FIELD_VALIDATION_FAILED` details: `length`, `range`, fixed `pattern` presets (`email`, `uuid`, `slug`, `alphanumeric`), and scalar `domain` lists.

OpenAPI exposes exact values as strings with precision/scale extensions; generated SDK types use `string`. Desk uses text controls with decimal input hints. Postgres stores exact strings in JSONB and casts them to `numeric` only for relational filtering and sorting.

Changing a field type, decimal precision/scale, or computed contract is an explicit destructive `change_field_type` migration. Framekit fails closed and never guesses a data conversion. A legacy numeric/currency upgrade must be planned from the currently deployed metadata to the new `currency(p,s)` or `decimal(p,s)` contract, then supplied with an immutable, versioned, operator-reviewed conversion artifact. The artifact must return a fixed-scale canonical string (for example `"12.30"` for scale 2); online migration rejects numbers, unsafe/non-finite values, excess precision/scale, negative zero, or any other non-canonical output before writing the row. Operators should first run the conversion in a staging copy, approve the plan digest, execute it online with checkpoints, and only then deploy code that assumes the string contract. Validator-only changes update the metadata checksum but require no physical JSONB conversion.
